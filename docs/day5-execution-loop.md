## Role: Execution Loop Agent

Ты автономный агент-исполнитель. Перед тобой пул задач. Ты берёшь задачу, определяешь её тип, выполняешь, коммитишь, переходишь к следующей. Человек НЕ вмешивается.

---

## LOOP WORKFLOW

Для КАЖДОЙ задачи:

```
1. CLASSIFY → определить тип задачи (bugfix / feature / refactor / docs / test)
2. LOAD PROFILE → прочитать соответствующий профиль из .claude/profiles/
   - bugfix → .claude/profiles/bugfix-profile.md
   - research → .claude/profiles/research-profile.md
   - feature / refactor / docs / test → использовать стандартный CLAUDE.md workflow
3. EXECUTE → выполнить задачу, следуя State Machine из CLAUDE.md:
   [ANALYSIS] → [PLANNING] → [EXECUTION] → [VALIDATION]
4. VALIDATE → обязательная проверка:
   - Запустить: python -m pytest tests/ -v --tb=short
   - Если тесты падают — чинить ДО коммита
   - Проверить: type hints, docstrings, соответствие инвариантам
5. COMMIT → git add + git commit -m "[loop] #N {краткое описание}"
6. LOG → записать результат в tasks/execution-log.md
7. NEXT → перейти к следующей задаче
```

**Правило остановки:** если на задаче зацикливаешься (>3 попыток чинить одно и то же) — запиши в лог причину, пропусти, иди дальше.

## Пул задач

Выполняй СТРОГО по порядку. Не пропускай, не меняй очерёдность (кроме зацикливания).

### #1 — Bug: /new_chat без активной conversation
**Тип:** bugfix
**Описание:** Если пользователь вызывает `/new_chat` когда нет активной conversation, должен корректно создаваться новый диалог без ошибки.
**Критерий:** edge case обработан, тест добавлен, pytest зелёный.

### #2 — Feature: typing indicator
**Тип:** feature
**Описание:** Добавить `bot.send_chat_action(ChatAction.TYPING)` перед вызовом LLM API. Пользователь должен видеть "печатает..." пока ждёт ответ.
**Критерий:** при отправке сообщения в Telegram видно статус "typing".

### #3 — Feature: разбивка длинных сообщений
**Тип:** feature
**Описание:** Telegram ограничивает сообщение 4096 символами. Если ответ AI длиннее — разбить на части и отправить несколькими сообщениями. Разбивать по абзацам (не резать посередине слова).
**Критерий:** функция `split_message(text, max_len=4096)` в `src/utils/text.py`, тест на разбивку, handler использует.

### #4 — Refactor: вынести строковые константы
**Тип:** refactor
**Описание:** Все хардкод-строки в handlers/ и services/ (приветствия, ошибки, fallback-сообщения) — вынести в `src/utils/messages.py` как константы.
**Критерий:** ни одной русской строки в handlers/services, все импортируются из messages.py.

### #5 — Bug: LLM ошибка ломает контекст
**Тип:** bugfix
**Описание:** Если LLM API вернул ошибку, user-сообщение уже сохранено в БД, но assistant-ответа нет. Следующий запрос отправит "дырявый" контекст. Нужно: или откатить user-сообщение при ошибке, или сохранять оба атомарно.
**Критерий:** при ошибке LLM user-сообщение НЕ сохраняется в контексте. Тест.

### #6 — Feature: /stats
**Тип:** feature
**Описание:** Команда `/stats` показывает: кол-во всех диалогов, кол-во сообщений, дата первого сообщения.
**Критерий:** handler + service + repo-метод. Ответ в Telegram форматирован.

### #7 — Feature: логирование времени LLM
**Тип:** feature
**Описание:** Логировать время ответа LLM: `logger.info("LLM response in %.2fs for user %s", elapsed, user_id)`.
**Критерий:** в логах видно время каждого запроса. Уровень INFO.

### #8 — Refactor: mypy clean
**Тип:** refactor
**Описание:** Добавить `py.typed` маркер в `src/`. Запустить `mypy src/` и исправить все ошибки типизации.
**Критерий:** `mypy src/` — 0 errors.

### #9 — Refactor: __all__ экспорты
**Тип:** refactor
**Описание:** Добавить `__all__` во все `__init__.py` файлы с явным перечислением экспортов.
**Критерий:** каждый `__init__.py` имеет `__all__`.

### #10 — Refactor: глобальный error handler
**Тип:** refactor
**Описание:** Зарегистрировать глобальный обработчик ошибок через `dp.errors.register()`. При необработанном исключении: логировать traceback, ответить пользователю "Произошла внутренняя ошибка."
**Критерий:** обработчик зарегистрирован, при raise Exception в service → пользователь получает сообщение, не видит traceback.

### #11 — Bug: MAX_CONTEXT_MESSAGES=0
**Тип:** bugfix
**Описание:** Если в конфиге `MAX_CONTEXT_MESSAGES=0`, запрос к LLM отправляется с пустым списком messages → ошибка API.
**Критерий:** валидация в config.py (минимум 1), тест.

### #12 — Feature: /history
**Тип:** feature
**Описание:** Команда `/history` — показать последние 5 сообщений текущего диалога (user + assistant), форматированных как:
```
👤 Привет
🤖 Привет! Чем могу помочь?
👤 Расскажи про Python
🤖 Python — это...
```
**Критерий:** handler + service, форматированный вывод.

### #13 — Docs: docstrings на services/
**Тип:** docs
**Описание:** Добавить docstrings на все публичные методы в `src/services/`. Формат: одна строка + Args + Returns + Raises.
**Критерий:** каждый публичный метод — docstring.

### #14 — Docs: CHANGELOG.md
**Тип:** docs
**Описание:** Создать `CHANGELOG.md` в формате Keep a Changelog. Описать текущую версию (0.1.0) со всеми реализованными фичами.
**Критерий:** файл создан, формат корректный.

### #15 — Docs: ADR-001 выбор aiogram
**Тип:** docs
**Описание:** Создать `docs/adr/001-aiogram.md` — Architecture Decision Record. Почему aiogram 3.x, а не python-telegram-bot или Telethon. Формат: Context → Decision → Consequences.
**Критерий:** файл создан, аргументация осмысленная.

## MUST DO

- Каждая задача — отдельный коммит с префиксом `[loop]`
- После каждой задачи — `pytest tests/ -v` (если есть тесты)

- State Machine из CLAUDE.md соблюдается на каждой задаче
- При обнаружении бага в процессе — записать, НЕ чинить (если это не текущая задача)

## MUST NOT

- НЕ менять порядок задач (кроме skip при зацикливании)
- НЕ объединять задачи в один коммит
- НЕ рефакторить код, если задача — bugfix (и наоборот)
- НЕ пропускать VALIDATION (особенно pytest)
- НЕ трогать задачи, которые уже сделаны в днях 3-4 (T1-T4, если уже есть)
