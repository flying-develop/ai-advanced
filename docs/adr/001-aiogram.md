# ADR-001: Выбор aiogram 3.x в качестве Telegram Bot Framework

**Статус:** Принято  
**Дата:** 2026-04-18

---

## Context

Для реализации Telegram-бота необходим Python-фреймворк, поддерживающий:

1. Асинхронный I/O — бот интегрируется с внешним LLM API (HTTP-запросы), SQLite (через aiosqlite) и Telegram Long Polling. Блокирующие вызовы недопустимы.
2. Middleware — необходима многоуровневая обработка каждого Update: аутентификация пользователя → инъекция сессии БД → инъекция сервисов.
3. Роутеры — логика команд (`/start`, `/stats`, `/history`) и обработки обычных сообщений должна быть разделена по модулям без глобального состояния.
4. Зрелость и поддержка — фреймворк должен активно развиваться и иметь документацию на рабочий API.

Рассматривались три варианта:

| Фреймворк | Версия | Async | Middleware | Роутеры |
|-----------|--------|-------|-----------|---------|
| **aiogram** | 3.x | ✅ native | ✅ built-in | ✅ Router |
| **python-telegram-bot** | 20.x | ✅ | ⚠️ limited | ⚠️ ApplicationHandlerGroup |
| **Telethon** | 1.x | ✅ | ❌ нет | ❌ нет |

---

## Decision

Выбран **aiogram 3.x**.

### Аргументы за aiogram 3.x

**1. Async-first архитектура**  
aiogram 3.x написан с нуля под asyncio. Нет legacy-слоя синхронного кода. Все внутренние операции — корутины, что критично для проекта с одновременными вызовами LLM API и БД.

**2. Гибкие роутеры**  
`Router` позволяет разбить обработчики на модули (`start_router`, `assistant_router`, `stats_router`) и регистрировать их декларативно. В `python-telegram-bot` аналогичное деление значительно громоздче.

**3. Полноценный middleware pipeline**  
aiogram 3.x поддерживает `outer_middleware` и `middleware` с явным порядком выполнения. Это позволило реализовать трёхуровневую цепочку: `AuthMiddleware` → `DbSessionMiddleware` → `ServiceInjectMiddleware` без глобального состояния.

**4. Dependency Injection через data-dict**  
Middleware передаёт зависимости через `data` dict, handler получает их как именованные аргументы. Это стандартный DI без сторонних IoC-контейнеров.

**5. Поддержка `ErrorEvent`**  
`dp.errors.register()` даёт централизованный обработчик необработанных исключений — важно для production-бота.

### Аргументы против альтернатив

**python-telegram-bot 20.x**  
- Добавил async в версии 20, но архитектура фреймворка изначально синхронная — это чувствуется в API.
- Нет понятия Router/middleware в смысле aiogram — нужно вручную группировать хэндлеры.
- Документация и сообщество меньше, чем у aiogram для async-сценариев.

**Telethon**  
- Это MTProto-клиент (user-mode), а не Bot API. Не подходит для классического бота.
- Нет встроенного механизма обработки команд, middleware, роутеров.
- Требует номер телефона, а не bot token — другая модель доступа.

---

## Consequences

### Положительные
- Чистая трёхслойная архитектура (Handler → Service → Repository) без обходных путей.
- Каждый Update обрабатывается в изолированном контексте (отдельная DB session, отдельный service).
- Легко добавлять новые команды: создать файл в `handlers/`, добавить `Router`, зарегистрировать в `bot.py`.

### Отрицательные / Риски
- aiogram 3.x имел breaking changes при переходе с 2.x; при мажорном обновлении придётся адаптировать middleware-слой.
- Сообщество меньше, чем у `python-telegram-bot` — часть Stack Overflow ответов устарела для v3.
- Документация aiogram 3.x местами неполная; для нестандартных сценариев нужно читать исходный код.
